# Architecture Decision Records

主要な設計判断のメモ。必要に応じて追記する。

## ADR-001: 技術スタックとホスティング
- **決定**: フロントエンドのみの SPA。Vue 3 + TypeScript + Vite を採用し、GitHub Pages でホスティング。
- **理由**: 依存が少なく、Pages での静的配信と相性が良い。TypeScript で型安全を担保。
- **影響**: サーバサイドは持たない。オフライン前提で機能を実装する。

## ADR-002: ルーティング構造
- **決定**: Vue Router を導入し、`/`（Home）と `/tools/:id` で各ツールを分割。ルートは遅延 import。
- **理由**: ツール数が多く、1ページ集約では初期バンドルが重くなる。モバイル向けに初期ロードを軽減。
- **影響**: ナビゲーションをヘッダー/メニューに集約し、共通レイアウトを整備する必要がある。

## ADR-003: モバイル優先レイアウト
- **決定**: 画面幅 360–430px を基準に 1 カラムで設計し、PC でもモバイル幅を維持。
- **理由**: 主用途がスマホであり、PCでもシンプルな表示で十分。レスポンシブ複雑化を避ける。
- **影響**: レイアウトの max-width を固定し、タップ領域や余白をモバイル基準にする。

## ADR-004: オフライン対応
- **決定**: Service Worker を利用し、全ツールがオフラインで動くことを前提とする。外部APIに依存しない。
- **理由**: ネットワーク不安定な環境でも使えるようにするため。GitHub Pagesでも実現可能な範囲。
- **影響**: データ取得はローカルの計算/定数に限定。SW と `base` の整合に注意する。

## ADR-005: 広告/トラッキングなし
- **決定**: 広告・トラッキングコードを組み込まない。
- **理由**: GitHub Pages での運用と非機能要件（趣味利用・プライバシー配慮）。
- **影響**: 収益化は考慮しない。計測が必要な場合はローカルのみの軽量な手段に限定する。
